# P2 — Agent-teams vs Workflow 모드 A/B (방법론 + 부분 실측)

> **작성:** 2026-06-07 · **상태:** 방법론 + 부분 데이터 전달. 완전한 통제 A/B는 미수행(아래 차단 원인).

## 목표
같은 작업을 **agent-teams 모드**와 **workflow 모드**로 실행했을 때의 비용(토큰)·시간·품질 델타를 측정한다.

## 왜 단발 실행으로는 정직한 숫자가 안 나오는가 (차단 원인)

1. **텔레메트리 비대칭 (핵심).** Workflow 도구는 실행 결과로 `agent_count`·`subagent_tokens`·`duration_ms`를 **네이티브 반환**한다. 반면 agent-teams는 teammate들이 여러 턴에 걸쳐 비동기로 돌고, 오케스트레이터에게 동등한 **per-run 합산 토큰/시간을 돌려주지 않는다**. 팀 비용을 잡으려면 세션 usage를 실행 전후로 diffing하는 **외부 회계**가 필요하다 → 단발 실행이 아니라 통제 실험.
2. **Experimental 플래그.** agent-teams는 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 의존(disabled by default). `TeamCreate` 스키마는 로드되나 런타임 가용·과금 환경은 세션마다 다를 수 있다.
3. **분산.** LLM 실행은 run마다 토큰이 흔들린다(테스트 가이드 §5-3 "고분산 eval"). n=1 비교는 신호가 약하다.

→ 이 세 가지 때문에 "한 번 돌려 X% 차이"는 **apples-to-oranges**가 된다. 가짜 정밀도 대신 아래를 남긴다.

## 보유 데이터 (실측, 이 세션)

| 항목 | 모드 | 에이전트 | 토큰 | 시간 |
|------|------|:--------:|------:|-----:|
| 런치 팬아웃/팬인 (`pilot_workflow`) | **workflow** | 4 | 168,291 | 197s |
| eval 하네스 2-suite (`eval_pilot`) | **workflow** | 16 | 282,633 | 34s |
| 원본 런치 런 (`_workspace/01~04`) | 모델 주도(팀/메인) | ~4 | **텔레메트리 없음** | 없음 |

→ workflow 쪽은 정밀 실측이 있으나, **팀 쪽 동일 작업의 텔레메트리가 없어 직접 비교 불가**.

## 문서화된 특성 (인용, research 보고서)
- agent teams: chat 대비 **~15배** 토큰, **teammate당 선형 증가**(각자 컨텍스트 창). 단일 에이전트 ~4배.
- 과투자 실패 모드: 단순 작업에 50 서브에이전트 스폰.
- workflow: 중간 결과가 **스크립트 변수**(컨텍스트 격리) → 대규모 팬아웃에서 메인 컨텍스트 오염 없음 + 네이티브 예산/텔레메트리.

→ **구조적 방향성**: 대규모·결정적·반복 작업에서 workflow가 유리할 *근거*는 충분(컨텍스트 격리 + 선형 비용 회피). 단, **특정 작업의 정량 델타는 미측정**.

## 정직한 통제 A/B 방법론 (다음에 실행할 절차)
1. **작업 고정** — 예: 4-role 팬아웃/팬인(auditor→content→scout→strategist), 동일 프롬프트·동일 모델.
2. **workflow 실행** — 결과 usage 네이티브 캡처. **N≥3회**(분산).
3. **agent-teams 실행** — `EXPERIMENTAL_AGENT_TEAMS=1` 켜고, 팀 비용은 **세션 usage 전후 diffing** 또는 `/cost`로 외부 회계. N≥3회.
4. **품질** — 산출물에 공통 루브릭(정확성/완성도/구조) 채점. 가능하면 블라인드 Comparator(테스트 가이드 §5-2).
5. **비교** — 토큰·벽시계·품질을 평균±분산으로. lift는 도메인별로 측정(이번 P3 eval 하네스가 with/baseline 채점 패턴을 재사용 가능하나, "모드 비교"는 별도 셋업).

## 결론
- **방향성(근거 충분):** 대규모 팬아웃·결정적 시퀀스·반복 실행에서 workflow가 비용·컨텍스트 측면 우위. (research + 구조적 분석)
- **정량(미측정):** 특정 작업의 % 델타는 텔레메트리 비대칭·플래그·분산 때문에 단발로 내면 오해. 위 통제 절차 + 외부 토큰 회계 + N≥3이 필요.
- **권장:** 실측이 필요하면 플래그를 켠 별도 세션에서 위 절차로 진행(예산 수반). 그 전까지는 "workflow가 대규모에서 유리(근거 O)·정량 델타 미측정(정직)"로 인용.
